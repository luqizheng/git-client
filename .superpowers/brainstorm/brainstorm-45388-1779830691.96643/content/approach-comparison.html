当前实现 vs pvigier 算法 — 差距分析

三个核心差距决定改造方案的选择

差距 1：排序算法

当前：纯时间排序
commits.sort((a,b) => a.time - b.time)

rebase/cherry-pick/时钟不同步时 → 边可能向下画，破坏布局

pvigier：Temporal Topological Sort
DFS + 按 committer date 排序遍历

保证拓扑正确 + 确定性 + 日期优先

差距 2：列分配逻辑

当前：两阶段启发式
1. 朴素继承父列
2. lastCommitInColumn 冲突检查 → 向右偏移

⚠ 不区分 parents[0] vs parents[1]
pvigier：Straight Branches
branch child (parents[0]) → 替换父列
merge child (parents[1]) → 不入列
active branch 列表管理

✓ 分支笔直不偏移

差距 3：颜色管理

当前：HSL 随机生成
hsl((col*137)%360, 70%, 55%)
无限色，但不同仓库色不同
pvigier：固定调色板
CSS 变量 8 色调色板
列回收后颜色可复用
✓ 视觉一致性好

改造方案选择

A

轻量改造：修复排序 + 列分配

改动：

  • 替换排序为 Temporal Topological Sort
  • 列分配改为 active branch 列表 + branch/merge child 区分

Pros

  • 算法完全对齐论文
  • 适应所有 Git 场景

Cons

  • 排序复杂度提升
  • 需构建 children 索引
B

全量改造：直分支 + 固定调色板

改动：方案 A + 固定 8 色调色板 + 颜色回收

Pros

  • GitKraken 级视觉质量
  • 颜色可复用/回收

Cons

  • 更多改动
  • 需处理边界颜色冲突
C

极简修复：仅修排序

改动：仅将排序改为 Temporal Topological Sort

Pros

  • 改动最小
  • 修复拓扑问题

Cons

  • 列分配问题仍存在
  • 复杂 merge 场景仍有异常
