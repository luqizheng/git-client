算法设计 — Temporal Topological Sort

第一步：确定行的顺序

为什么要拓扑排序？

纯时间排序在 rebase/cherry-pick 后父子关系与时间戳冲突，导致边向下画。

输入输出
输入: commits[].committer_time   输出: rowIndex (0=新/顶, N-1=旧/底)

步骤:
1. 从 parent_ids 构建 children 索引  Map<commitId, Commit[]>
2. 按 committer_time 降序排列（最新优先）
3. DFS 遍历，后序赋 rowIndex：
   - dfs(c) → 先 dfs 所有 children → 再 c.rowIndex = i++
4. 结果：始终满足 parent.rowIndex > child.rowIndex

示例：rebase 异常场景

纯时间排序 ❌
A ← B (rebase 后, 时间较旧)
      ↑ 边向下！╳
拓扑排序 ✓
B (newer)   rowIndex=0
A (older)   rowIndex=1
↑ 边始终向上 ✓

算法设计 — Straight Branches 列分配

第二步：确定每列的编号

核心概念：branch child vs merge child

基于 parents[0] 区分

branch child: child.parents[0] == c — 延续分支的子提交

merge child: child.parents[0] != c — 合并进来的子提交

直分支规则：

  • branch child 与父提交同列（replace）
  • merge child 使父提交从 active branches 移除
  • 非 branch child 的分叉创建新列（append到最右）

列分配过程

activeBranches = []  // {commitId, column}

for each c in commits FROM NEWEST TO OLDEST:
  // c 可能已由它的 branch child 占据了一列
  // 也可能是一个合并目标
  branchChildren = children where child.parents[0]==c.id
  mergeChildren = children where child.parents[0]!=c.id

  if c has branchChildren:
    selected = pick leftmost branch child
    // 取该 child 的列，c 继续使用
    c.column = selected.column
    activeBranches[selected.column] = c  // replace
  else:
    // 分支终点或新分支
    if c has mergeChildren:
      // c 被合并进来了，使用 merge child 的列
      c.column = mergeChildren[0].column
    else:
      // 新分支终点，追加新列
      c.column = len(activeBranches)
      activeBranches.append(c)
  
  // 从 activeBranches 移除 mergeChildren
  for child in mergeChildren:
    activeBranches[child.column] = nil
✓
关键改进：列分配方向改为 从新到旧（而非当前的从旧到新），与 pvigier 一致
