🔴 关键风险（必须修复）

R1: Phase 1 纯 UI 无功能增量 — 用户无感知

问题：Phase 1 仅改色系和视觉打磨，不增加任何功能。对用户来说「更新了但没什么不同」。

建议：Phase 1 应与 Phase 2.4（Undo/Discard）并行交付。Discard 是当前最痛的缺失（StagingPanel 直接提示"不支持"），且不依赖 Diff 视图。这样 Phase 1 结束时用户能感知到功能变化。

修订：Phase 1 = UI 统一 + Undo/Discard + Cherry-Pick 右键入口

R2: 交互式 Rebase 技术复杂度被低估

问题：拖拽排序 + 操作切换 + 冲突自动跳转，涉及 git2 的 rebase API 和复杂状态机。GitKraken 团队花了好几版才做好。

建议：MVP 版本先做「列表式交互 Rebase」（无拖拽，用上下箭头调整顺序），而不是一步到位做拖拽。拖拽作为 P1 增强后续迭代。

修订：Rebase MVP = 列表式操作 + 上下箭头排序 + squash/reword/drop 切换

R3: 三方冲突解决缺少降级方案

问题：三方 diff 依赖 Monaco Editor 的 3-way merge 能力，但 Monaco 原生只支持 2-way diff。需要自建三方视图或用第三方库。

建议：MVP 先做「2-way + 手动编辑」模式（Ours vs Theirs，冲突区域高亮，底部编辑区手动合并），三方视图作为增强版迭代。2-way 已经能解决 90% 的冲突场景。

修订：冲突解决 MVP = 2-way diff + 手动编辑区 + Mark Resolved

🟡 值得关注的问题

W1: Undo/Redo 基于 reflog 有边界风险

reflog 是本地操作记录，但不是所有操作都能安全 undo（如已 push 的 commit undo 后会 diverge）。需要明确哪些操作允许 undo，哪些需要警告。

建议：增加「已推送操作 undo 前强制确认」规则

W2: 设置面板 9 个 Tab 同时实现范围偏大

Phase 3 包含 9 个设置分类 + 6 个新页面，是三个 Phase 中工作量最大的。但设置面板不影响核心功能使用。

建议：Phase 3 拆分为 3a（Git Config + 通用增强）和 3b（Proxy + Editor + Shortcuts + Hooks），3a 先交付

W3: CSS 变量迁移可能遗漏硬编码颜色

组件中可能存在硬编码颜色值（如 Sidebar.vue 的 `border-gray-700`、`hover:bg-gray-800/50`），仅改 CSS 变量不够。

建议：Phase 1 增加全局扫描，将所有硬编码颜色替换为 CSS 变量或 UnoCSS 语义 token
